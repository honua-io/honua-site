Why gRPC

Three things REST can't give you cleanly.

Typed contracts

Protobuf schemas instead of guessing at JSON.

Every operation has an explicit input and output shape. Clients are generated, not written by hand. Field evolution is versioned in the schema, not negotiated over Slack.

Throughput

Binary payloads for high-volume traffic.

App backends, integration services, and mobile sync get a leaner wire format than JSON or XML — meaningful when you're moving features and tiles at scale.

Streaming

Long-lived connections for sync and events.

Feature change streams, telemetry, and event-heavy workflows use native streaming instead of polling loops bolted onto REST.

Coexistence

gRPC and standards ship in the same binary.

You don't stand up a second stack. A WFS client and a gRPC client point at the same service, back the same data, and authenticate against the same auth.

Open protocol

The wire format is public.

The Protobuf schemas are on GitHub and going through the OGC submission process. Use them whether you run Honua or not.

geospatial-grpc

Proto definitions for feature, map, tile, and form services.

Read the schemas at honua-io/geospatial-grpc. Generate a client in any language with protoc.

OGC submission

Shaped as a standard, not a proprietary API.

The aim is for typed spatial APIs to become something any vendor can implement — the same way WFS and WMS are today.
