Why now

Spatial data used to live with the GIS team. Now every team is a spatial team.

Policy wants answers. Product wants maps. Operations wants routing. Field wants offline sync. Support wants territory views. Developers want typed SDKs. Platform engineers want something that runs on Kubernetes. AI agents want a protocol. None of today's tools were built to serve all of them at once. Honua is.

Who Honua is for

Five user classes. One foundation.

Spatial work isn't a single job anymore. Honua gives every role — specialist and non-specialist alike — a first-class way in.

01 · GIS Analysts & Leads

Offload the routine. Own the hard work.

Your ArcGIS Pro, QGIS, and GeoServer workflows keep running through the compatibility bridge. Agents and automation take the repetitive queries off your plate so you can spend your time on the spatial analysis, data stewardship, and domain judgement only you can do. Honua is built to make you the highest-leverage role in the room — not to work around you.

02 · Non-GIS Teammates

Ask the question. Get the answer.

Planners, councilors, operators, support leads — ask what parcels shifted, what routes miss coverage, which sites failed inspection. No license required, no ticket to file.

03 · Developers

Typed contracts in your language.

JavaScript, Python, .NET, MAUI mobile. gRPC under the hood. Offline-first sync for field apps. One client surface across web, backend, desktop, and device.

04 · Platform Engineers

A GIS that ships like your other services.

Services defined in Git. Rollouts reviewed in PRs. OpenTelemetry traces through the runtime into PostGIS. Six validated deploy targets on AWS and Azure.

05 · AI Agents

A protocol, not a parser.

MCP-native access to every collection, schema, operation, and topology tool. Claude, GPT, Cursor, or your own agent — no scraping, no ad hoc REST, safe by construction.

02 · SDKs for everything

Your developers ship in the language they already use.

Typed contracts. Offline-first sync. One client surface across web, backend, desktop, and field. Pick the runtime that fits the team and the job.

JavaScript web · node 
import { HonuaClient } from "@honua/sdk";

const c = new HonuaClient("https://gis.example.com");
const parcels = await c.features("parcels").query({
  where: "status = 'active'",
  bbox: [-122.42, 37.77, -122.40, 37.79],
});
Python analysis · pipelines 
from honua import Client

with Client("https://gis.example.com") as c:
    parcels = c.features("parcels").query(
        where="status = 'active'",
        bbox=(-122.42, 37.77, -122.40, 37.79),
    )
.NET services · desktop 
using Honua;

await using var c = new HonuaClient("https://gis.example.com");
var parcels = await c.Features("parcels")
    .Where("status = 'active'")
    .InBbox(-122.42, 37.77, -122.40, 37.79)
    .ToListAsync();

05 · Operate like software

Run GIS the way you run the rest of your infrastructure.

Services defined in Git. Rollouts reviewed in PRs. Traces that reach from the edge into PostGIS. An agent on call so your team doesn't have to be.

GitOps

Publish layers, feature services, and styling from YAML in a repo. Review changes in PRs. Roll back with git revert.

Watches logs, triages pages, proposes fixes as pull requests. Your oncall reviews; it doesn't draft from scratch.
