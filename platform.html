Platform

A full-stack geospatial platform, not another feature server.

Honua starts with server replacement and grows into the platform for data services, apps, mobile, analytics, agent access, and managed delivery.

Differentiation

Migration is the beachhead. Platform consolidation is the play.

The strongest wedge is replacing legacy GIS without breaking clients. The larger opportunity is to unify serving, field workflows, analytics, automation, and operations on one contract set.

Compatibility

Keep the old clients. Fix the operating model.

GeoServices REST and OGC compatibility make migration credible. Buyers do not need to rip out every client on day one to move to a better platform.

  • Service import gets data and services into Honua quickly.
  • Migration tooling and codemods reduce rewrite cost for app teams.
  • Staged cutover keeps legacy clients alive while new paths move to SDKs, gRPC, and MCP.

New Rails

Give the next generation of apps better interfaces.

gRPC and MCP are not side features. They are the forward path for application builders, mobile teams, and agent-native geospatial workflows.

Visual

The current cloud-native deployment spine.

Honua already ties clients, ingress, runtime, PostGIS, caching, storage, and monitoring into one deployment model instead of treating GIS as a one-off island in the stack.

Current deployment reference: one cloud-native runtime behind standard ingress, backed by PostGIS, optional Redis, object storage, and observability.

Platform Spine

The layers that make the full stack possible.

Runtime

Compatibility and serving core

Serve GeoServices REST, OGC API, OData, tiles, WMS, WMTS, unary gRPC, and MCP-backed discovery/query access from one PostGIS-backed system.

Admin

Admin APIs and control center

Publishing, governance, and rollout coordination belong inside the platform, not in scattered scripts and runbooks.

Integrations

SDKs, gRPC, and agents

JavaScript, .NET, Python, migration tooling, and MCP give developers a faster path from evaluation to production.

Mobile

Field workflows

The MAUI-first mobile stack brings offline sync, forms, AR-assisted workflows, and sensor-aware collection into the same platform.

Data

Data services and modern formats

Geocoding, routing, analytics, GeoParquet, FlatGeobuf, PMTiles, and COG belong on top of the same platform contracts.

Cloud

Self-hosted or managed

Docker, Kubernetes, AWS, Azure, serverless targets, and managed cloud can all share the same product spine.

Platform Direction

Where the stack expands next.

Area What Honua is building toward Why it matters
Serving and compatibility Protocol runtime, migration destination, standards surface, admin API, and deployment target It is the lowest-risk path out of legacy GIS and the foundation every other surface builds on.
Mobile and field operations Offline sync, forms, background upload, AR-assisted workflows, IoT and sensor-aware collection Most GIS platforms still stop at serving. Honua is pushing the stack all the way into field execution.
Data services Geocoding, routing, spatial analytics, H3-style aggregation, and streaming data flows That is how Honua becomes a broader geospatial application platform, not just a serving runtime.
Formats and storage GeoParquet, FlatGeobuf, PMTiles, COG, and modern data movement patterns Cloud-native geospatial stacks need modern file and archive formats, not just database tables and shapefiles.
Apps Portal, dashboards, open data hub, and low-code geospatial application surfaces The platform can own more of the user-facing stack instead of stopping at APIs.
Managed cloud and operator tooling Honua Cloud, deployment orchestration, rollout evidence, and partner or enterprise operator workflows Buying GIS as a platform is easier when self-hosted and managed paths share the same core contracts.
AI and automation AI spatial agent, geoprocessing, GeoETL, and private operator copilot flows Honua can expose geospatial workflows to both users and agents without giving up governance boundaries.

Why the platform strategy matters

GIS buyers are tired of stitching together one server, one tile service, one field app, one analytics tool, and one licensing conversation for each.

Why this is credible

The same shared runtime, admin contracts, SDKs, pricing model, and deployment substrates make the larger platform direction believable instead of aspirational fluff.
