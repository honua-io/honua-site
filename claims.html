// status vocabulary

How status labels are used.

Source-backed
Claim can be stated as Preview-current when linked evidence supports the exact scope.
Preview partial
Claim can be marketed for evaluation, but partial parity and caveats must stay visible.
Proof pending
Claim may be described as a Preview target, not as validated or generally available.
Roadmap / Beta
Claim belongs to Beta, named pilot, or later roadmap scope.
Deferred
Claim must not be marketed as Preview-ready.
External owner
Release gap belongs to another repo; this site only links the owner ticket or source.

// evidence bar

Self-imposed rules for everything above.

01 · Exact numbers
Use specific counts only where a CITE snapshot, parity JSON, or test artifact says the same number.
02 · Partial parity stays partial
Never describe GeoServices REST coverage as "100% parity" or "drop-in for everything." Gaps stay tracked in the matrix.
03 · No unconditional cutover
"Cut over when ready" / "clients keep working" need an attached parity run for the target estate.
04 · Roadmap labels stay visible
Beta, pilot-scoped, private-beta, and proof-pending labels stay on every page where the underlying claim sits.
05 · External owner = link
If another repo owns the gap, link the owner ticket. Don't restate the work as if it ships from this site.
06 · Examples are examples
Terminal sessions, transcripts, and code diffs that are illustrative get labeled "example" — not "live run."
07 · Benchmarks travel with methodology
Performance numbers cite the GeoBench snapshot or stay shape-only with the methodology linked.
08 · Pricing TBD stays TBD
Until commercial tiers are real, this site says "shaped in the open" — never invents a number.
