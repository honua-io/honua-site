Migration path

Lower switching cost with compatibility and checks.

Path Migration support Adoption value
ArcGIS service import Import FeatureServer, MapServer, metadata, schemas, styles, permissions, and service references as part of the migration scope. Existing ArcGIS-facing clients can be validated against Honua before cutover.
GeoServer service import Import WFS, WMS, WMTS, WCS, layers, styles, CRS, metadata, and service endpoints as part of the migration scope. GeoServer-facing clients keep using familiar standards while the backend changes.
Standards and data ingestion OGC services, GeoServices REST, files, databases, and object storage paths support staged movement. Teams can move service by service and dataset by dataset.
SDK and app converters Migration helpers can identify repeatable ArcGIS JS, SDK, layer config, style, form, and service-reference patterns. Mechanical app migration work gets automated, while hard judgement calls stay visible.
Parity checks Compare features, tiles, rendered maps, response shapes, and workflow behavior side by side. Teams can prove equivalent behavior before users are moved.

Migration model

Run old and new together.

The intended adoption path is side-by-side: keep legacy clients on standards and compatibility surfaces, move new work to SDKs, gRPC, and agents, then cut over at the pace the organization can support.

1. Import

Bring existing services forward.

Start with ArcGIS, GeoServer, files, and database-backed services already in production.

2. Validate

Check parity before rollout.

Compare outputs and fix differences before changing user traffic.

3. Expand

Move new work to modern interfaces.

Use SDKs, gRPC, MCP, GitOps, and AI workflows after the first services prove out.
